The point of this deck: explain the architecture, the data flow, and why the module is built page by page rather than as one giant batch.

Docling for extraction and OpenAI vision for semantic descriptions of visual elements.

Slide 4

Internal Building Blocks

Docling Converter

Runs OCR, table extraction, formula enrichment, page rendering, and image extraction from the PDF.

Asset Collectors

Walk the parsed document and separate pictures, tables, and formulas into dedicated buckets per page.

Markdown Renderer

Replaces placeholders with image links and generated prose so each page becomes retrieval-ready markdown.

Configuration choices

  • OCR is enabled so scanned or image-heavy PDFs still produce usable text.
  • Table and formula extraction are enabled because those elements matter for retrieval.
  • Page, picture, and table images are all rendered for later reuse.

Architecture implication

  • The processor is doing both structural extraction and semantic enrichment.
  • It converts visual knowledge into searchable text without losing image references.
  • Every downstream stage inherits the page alignment established here.
Three major zones: extract, enrich, assemble That is the mental model to explain on video
Slide 5

Why It Runs Page by Page

Execution loop

Count pages

Find the start and end page to process.

Convert one page

Pass a single-page range into the converter.

Cleanup

Release memory before moving to the next page.

Why this shape is useful

  • Limits peak memory and keeps the pipeline stable on modest hardware.
  • Makes page numbers explicit and reliable throughout the pipeline.
  • Produces per-page markdown files that are easy to inspect and debug.

Operational benefit

Failures are isolated to a page instead of taking down the whole document.

Retrieval benefit

Downstream chunks can keep clean page provenance.

Content benefit

Images, tables, and formulas remain tied to the page where they appeared.

Architecture choice: stability over raw throughput That is the key reason this module feels different

Important architecture detail

The system preserves asset order as it walks the page, so the final markdown can reinsert the right description in the right place.

Why this matters for RAG

RAG systems retrieve text, not raw visual intuition. These descriptions convert images and tables into searchable knowledge while still keeping the original asset files around.

Slide 7

How Final Markdown Is Assembled

Raw extraction

Docling first exports page markdown with placeholders for images.

Normalization

Tables and display formulas are converted into custom placeholders so every asset type follows the same replacement pattern.

Page markdown

Text with temporary placeholders.

Placeholder scan

Walk left to right through the page content.

Asset replacement

Insert image references and generated descriptions.

Page block

Wrap the result with a `--- PAGE n ---` marker.

The output contract downstream depends on

  • Stable page markers for page-aware chunking.
  • Relative asset paths that point to saved images.
  • One stitched markdown file that preserves reading order across the whole document.
By the end of this stage, the PDF has effectively been rewritten as enriched markdown That markdown becomes the canonical text source
Slide 8

Downstream Handoff: Chunking to Answers

Chunking Layer

Reads `processed_doc.md`, keeps page boundaries, and can peek at the next page to preserve cross-page meaning.

Embedding Layer

Embeds each refreshed chunk and stores it in a persistent Chroma collection with page metadata.

Retrieval Layer

Rephrases the user query, searches similar chunks, and collects associated page numbers.

Answer Layer

Combines retrieved text with page screenshots to generate grounded answers in the Streamlit app.

Files passed downstream

processed_doc.md page_images/ chunks.json Chroma collection

Main message for the audience

`doc_processor_2` does not answer questions itself. It prepares the document so the later RAG stages can retrieve and answer accurately.

The architecture is sequential but cleanly staged Each layer consumes a concrete artifact from the previous one

The 30-Second Summary

What it is

An ingestion-and-enrichment module for RAG, not a general PDF viewer or chatbot.

What it does

Turns a PDF into page-aware markdown plus searchable descriptions of visuals, tables, and formulas.

Why it matters

It gives the rest of the RAG pipeline structured, grounded inputs instead of messy raw document data.
